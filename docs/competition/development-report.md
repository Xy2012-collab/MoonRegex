# MoonRegex 开发报告

## 开发过程

本项目从零构建了一个纯 MoonBit 正则表达式引擎。采用 AST（抽象语法树）编译 + 递归回溯匹配的架构，避免了 NFA/DFA 构造的复杂性，在 MoonBit 的函数式编程约束下保持了代码的简洁性。

开发分为四个阶段：
1. **AST 设计**：定义 RegexAST 枚举（Lit/Dot/Star/Plus/Opt/Alt/Seq/Class/Escape/Anchor/Empty）
2. **解析器**：实现递归下降解析器，将正则字符串编译为 AST
3. **匹配引擎**：实现基于 AST 的递归回溯匹配器，支持全部量词和组合
4. **API 完善**：添加 match/find/find_all/replace/split/count/match_result 等公共接口

## 架构设计

```
正则表达式字符串
  → Parser（递归下降）→ RegexAST（AST树）
  → Matcher（递归回溯）→ 匹配结果
```

核心类型：
- `RegexAST` 枚举：12 种节点类型，覆盖完整正则语法
- `Regex` 结构体：编译后的正则对象，提供全部 API

匹配引擎采用递归回溯策略：
- `Star` 通过贪心尝试最长匹配，失败时逐字符缩短
- `Alt` 先尝试左分支，失败回退到右分支
- `Seq` 顺序匹配，任一步失败则整体失败

## 技术难点

### 1. 贪心量词的回溯限制
`match_star` 实现了贪心匹配：先尝试尽可能长的匹配，逐步缩短。但对于 `a.*z` 匹配 `a123z`，`.*` 的贪心策略会消耗掉末尾的 `z`，导致回溯路径探索不充分。这是递归回溯实现的已知局限——完整的回溯需要保存更多状态。

### 2. 字符类范围展开
MoonBit 没有 `Char.to_int()` 等内省方法，字符范围 `[a-z]` 需要手动实现 `char_to_int` 和 `int_to_char` 映射表，通过 if-else 链覆盖 ASCII 字母和数字，并使用递归函数 `range_add` 展开范围到字符数组。

### 3. 字符串不可变性
匹配器的 `replace` 函数需要拼接大量字符串。MoonBit 不支持可变字符串累加，解决方案是使用递归函数 `replace_rec` 通过参数传递累加结果，每次递归创建新的字符串。

### 4. 枚举变体语法
MoonBit 0.9 枚举使用位置参数而非命名参数：`Lit(String)` 而非 `Lit(s : String)`。枚举模式匹配使用 `RegexAST::Lit(s)` 解构第一个位置参数。

### 5. `char` 关键字冲突
最初尝试使用 `char` 作为 State 枚举的字段名，发现 `char` 是 MoonBit 保留关键字，改为 `ch` 后编译通过。

## 不足与展望

1. 回溯实现不支持 `a.*z` 这种需要非贪心匹配的模式
2. 不支持捕获组（capturing groups）的内容提取
3. 不支持 `\b` 词边界和 `\B` 非词边界
4. `$` 锚点未强制执行行尾检查
5. 字符范围仅覆盖 ASCII 字母数字，不完整
